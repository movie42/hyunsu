작년 말미부터 개인적으로 너무 많은 일들이 있었다. 게다가 정말 끔찍한 일을 경험 할 뻔했다. 사람이 많이 죽을 뻔했다. 정신적으로 많이 힘들었다.  
그래서 그동안 있었던 일을 다 나열하기가 어렵다. 혼란스럽고 두려운 시간을 지나고 '다행이다.' 마음을 쓸어담으면서 서울 재즈 페스티벌을 갔었다.
그게 벌써 5월이다. 같이 개발 이야기를 주고 받았던 사람들과 왕래가 없어진지 오래고 이제 거의 개발 이야기를 하지 않는다. 대신 AI와 하루 종일 붙어있다보니
이제 얘가 없으면 나는 어떻게 하나 하는 생각 뿐이다.

# 화이트레이블

이번 해부터 약간 회사의 경영 구조 변경으로 SI 비스므레한(완전한 SI가 아니라) 프로젝트를 계속 하게 되었다. 이전에도 나는 계속 외부 업체 커스터마이징이나
외부 비즈니스로직이 회사 프러덕트 비즈니스 로직에 영향을 주지 않으면서 쉽게 추가하거나 덜어내는 작업을 많이 했었다.
그간의 경험 덕분인지 화이트레이블 프로젝트를 담당하게 되었다. 이 프로젝트의 목표는 최대한 인력 구성없이 외부 업체 요구사항을 최대한 빠르게 커스터마이징 할수 있는
환경을 만드는 것이다. 집에 가기전에 팀장님과 이야기를 나누다가 '할수 있겠는데.'하는 생각이 들었다.

처음엔 그냥 복붙 하라고 해서 투덜대면서 했다. 하지만 내부의 코드를 개선하고 싶어도 엄두를 낼수 없는 것들이 많았다. 간단해 보이는데 실상은 까보면 원래대로 동작하게 하면서
내부만 수정하는건 정말 어려운 일이었다. 그래서 테스트 코드가 필요하다고 생각했다. 하지만 테스트 코드도 결국 코드다. 작성 후 지속적으로 개선을 해야하고 코드에 새로 추가되는
부분에 동일하게 추가되어야한다. 또 테스트가 코드가 변경될때 자동으로 테스트를 하면서 지속적으로 개발자에게 유지 보수를 요구해야한다.
하지만 이전에 자동화를 여러 시도가 있었지만 모종의 이유로 테스트를 수동으로 할 수밖에 없었다.
그리고 테스트 코드가 모든 구멍을 다 막아주는 것은 아니다. 테스트 코드는 제품 운영을 위한 일부분중 하나다.
결국 회사 내에 타 부서와 협의를 해야하는 부분도 있고 현재 진행되고 있는 프로젝트의 일정을 고려하다보면 리펙토링은 계속 후순위로 밀릴 수 밖에 없다.
레거시가 꼭 나쁘다고 말할수 없다. 레거시는 레거시로써 feature의 한 축을 맡아 돌아가고 있다. 그런데 그걸 드러내고 새로운 코드를 이식하는 일은 '이미 바쁜데 뭐하러?'라는 생각이 들수밖에 없다.

어쨌든 이런 저런 여러 이야기 뒤에 새로운 A 프로젝트를 진행하면서 모노레포를 처음으로 프러덕트 별이 아닌 도메인별로 구성하게 되었고 이 프로젝트에서 여러가지를 마이그레이션하게 되었다.

```
Redux Pender => Redux Toolkit
React Router Dom v5 => React Router Dom v7
React Query => Tanstack React Query
luxon, moment => date fns
```

마이그레이션 하면서 React Router Dom v7으로 라우터를 변경하게 되었는데 그렇게 되면서 라우터 생성을 이전보다 단순하게 그리고 더 명확하게 책임을 분리할 수 있었다.
사실 비즈니스 요구사항에서 매장 정보가 가장 중요한데 이 정보를 불러오기도 전에 라우터가 한번 생성되서 화면이 그려지고 그 다음에 매장 정보를 불러오는 로직이 문제가 된적이 있었다.
먼저 매장에서 어떤 옵션을 사용하고 있는지를 보고 그 다음에 GNB를 구성하거나 아니면 완전히 다른 화면으로 리다이렉트를 해야하는데 그게 현재 구조에서 구현하기 매우 까다로웠다.
어쨌든 React Router Dom v7으로 변경하면서 이전보다 Authorization을 더 명확하게 그리고 더 쉽게 구현을 할 수 있었다.

# 내가 어떤 라이브러리를 어떻게 사용하는지가 중요하다

회사 동료 P님이 만든 useAdvancedSearchReducer는 DX가 정말 좋았다. 기존에 OptionGroups는 성능과 DX가 둘다 좋지 않았다.
query string을 넣기 위해서 OptionGroups에서 useEffect를 사용하고 있었는데, 그런데 이게 문제가 시간이 지나면서 여러 사람의 손을 타다보니 코드가 변형되면서
처음에 의도했던것과 다르게 동작하기 시작했다. 그래서 이 에러를 해결하기 위해 사용하고 있는 곳으로 가면 history.push가 수직으로 지뢰처럼 이곳저곳에 깔려있어서 디버깅이 죽을 맛이었다.
filter라는 option을 먼저 가져온 뒤에 UI가 랜더링되어야하는데 UI가 먼저 랜더링 되고 filter를 불러온 뒤에 filter 정보를 사용해서 UI를 만들고 그 filter
정보를 query string에 밀어 넣기 위해서 useEffect를 쓰고 있었다. 그러다보니 가끔 filter를 다시 불러와도 ui가 업데이트 되지 않거나 업데이트를 위해 다른 무언가를 하면
무한 호출이 발생했다. OptionGroup 문제를 해결하기 위해서 천을 덧대서 다시 기우다보면 코드가 해지다 못해 못쓰게 되었다. 나는 이걸 정말 건드리고 싶지 않았다. option을 제어하는 곳이라면
대부분 다 이것을 사용하고 있어서 버그 리포트가 슬렉에 올라오면 정말 '누가 제발 먼저 대답해줘'하고 메시지를 몇초간 외면한적도 많았다. 정말 죽을 맛이었다.

그래서 이 문제를 해결하기 위해서 useAdvancedSearch라는 훅을 만들었는데 이 훅의 가장 큰 단점은 query-string 상태를 ContextAPI 안에서 직접 관리하게 한 부분이었다.
이전과 다르게 무한 호출은 일어나지 않지만 query-string을 초반에 default 값으로 밀어 넣은 다음에 helper 함수로 Context 내부 query string의 상태를
직접 관리하게 하였다. 하지만 나는 관리를 하고 query의 상태를 사용하는 것은 외부에 위임했는데 나름 그 방법도 나쁘지는 않았지만 그걸 Context로 해결하려고 했다는 점이 조금 아쉬웠다.
차라리 react-router-dom을 활용하는 편이 더 낫지 않았을까? 결국 이 useQueryString이라는 ContextAPI 함수가 query 함수 내부에서 사용되고 있어서 자유도가 떨어졌다.

시간이 지나서 P님이 만든 useAdvancedSearchReducer가 만들어졌다. 얘는 애초부터 tanstack/react-query API를 활용해서 만들어진 훅이었는데 query string을
관리하는 방법이 놀랍도록 편리했다. 개발자가 처음에 params를 default로 밀어 넣기만 하면 그 이후는 actions, execute 두 함수를 사용해서 url 변경을 보류하거나 실행할 수 있었는데
회사 프러덕트 filter의 정책과도 정확하게 일치했다. 코드 내부에서 useReducer로 상태를 관리하고 있고, actions와 execute의 차이는 query string의 새로 업데이트하느냐 사용하느냐의
차이였다. useQuery는 query key가 변경되면 API를 새로 호출하기 때문에 useLocation을 사용하기만 하면 굳이 Context나 전역 상태 관리 도구에서 query string을 관리할 필요가 없이
API를 간편하게 호출할 수 있었다.

useAdvancedSearchReducer도 기본적인 아이디어는 나와 동일하다. query의 상태를 관리하는 것이다. 하지만 useAdvancedSearchReducer는 stateless하게 동작했고
내가 만든 useAdvancedSearch는 상태를 계속 신경써주어야했다. 이번에 useAdvancedSearchReducer로 전부 교체하고 deprecated 된 useAdvancedSearch는 사라졌다.

게다가 나는 이 훅을 사용해서 OptionGroups를 새로만들 수 있었다. Factory 패턴을 사용해서 AI와 함께 컴포넌트를 구현을 했다. 결국 검색 필터링 기능을 구현하는데 가장 어려운 점은
UI와 helper 함수가 따로 놀면서 그 싱크를 맞추기 위한 것이라고 생각한다. 하지만 useAdvancedSearchReducer 내부의 query string 상태를 변경하는 방법은 이전에 말했다시피 두개로 나뉘어있었고
Factory 내부에서 useAdvancedSearchReducer를 통해 url을 변경하도록 만들었기 때문에 Factory 내부에서 랜더링된 UI는 별다른 onChange 함수를 외부에서 필수로 주입할 필요가 없게 되었다.

# 선언 vs 명령

지겹도록 들어온 말이지만 사실 구현 할때 선언적으로 구현하는것은 쉽지 않다. 선언적 구현이 조금더 개발자에게 친화적이라는 말을 많이 들었지만 그렇게 구현하는 것이 쉽지 않다. 꼭 선언적이어야만 좋다는건 잘 모르겠다면 지금까지 구현한 UI가 얼마나 재사용이 어렵고 나중엔 예외 케이스를 소화하기 위해 props가 점점 많아지다 보면 조금씩 이 불편함을 해소하기 위해서 어떤 방법을 쓰는 것이 좋은지 고민하게 된다.

처음에 구현했던 Factory는 객체를 생성하는 helper 함수가 있었다. 이것도 나름 선언적이지만 문제는 react 스럽지 않다는데 있었다.

그래서 UI 영역과 도메인 영역을 나누는데 중점을 뒀고 구현했던 것을 다시 선언적으로 구현하기 시작했다. 이렇게 바꾼데는 UI를 만들기 위한 페이지를 많이 할애하게 된다는 것이 조금 이상하다는 생각이 들어서였다. 뭐 어쨌든 ...
